Key Engineering Challenges & Solutions

Technical decisions that enabled reliable operation.

Trajectory Planning: Cubic Splines

Challenge: Linear interpolation between waypoints produces discontinuous accelerations, causing motor overshoot and erratic behavior.

Solution: Cubic polynomial interpolation with zero-velocity boundary conditions. Solves a 4×4 linear system for each waypoint transition to guarantee continuous acceleration profiles.

Implementation: GimmeCoeff() computes spline coefficients; TargetDist() evaluates the cubic in real-time for each motor every 5ms. All 24 waypoint transitions use this approach.

Result: Smooth, predictable motion with minimal overshoot. Trajectory tracking error reduced to 5-8% across all axes.

Multi-Motor Cascade Control

Challenge: Five motors with disparate dynamics (3 linear, 2 rotational). A single PI/PID gain set cannot handle the variation.

Solution: Independent PI/PID loops for each motor with empirically tuned gains. Joints 1-3 use PI control; Joints 4-5 use PID for improved stability on the rotational axes.

Implementation: Each motor's error accumulates in esum, driving the integral term. Proportional term uses encoder-measured position. PWM output saturates at 0-100% duty cycle.

Result: All motors track their spline-generated setpoints with stable, bounded responses. System handles load transients without oscillation.

PWM Stability: Eliminating Jitter via DMA

Challenge: Initial implementation used software-timed PWM via Raspberry Pi GPIO. OS context switching introduced timing jitter, causing controller instability and poor encoder synchronization.

Root Cause Analysis: Oscilloscope measurements revealed PWM edges drifting ±2-5ms. This jitter propagated through encoder ISR contention, creating cascading control errors.

Solution: Switched to hardware-based DMA-timed PWM using the Raspberry Pi's PWM peripheral. Timer clock generates pulses independent of CPU load, ensuring deterministic timing.

Result: Zero timing jitter. Controller stability improved dramatically. System can now sustain 24-waypoint sequences with predictable, repeatable performance.

Takeaways

What Worked

  • Cubic spline trajectories provided smooth, predictable motion without overshoot.
  • Empirical gain tuning via step response enabled stable multi-motor coordination.
  • Closed-loop encoder feedback gave real-time control authority and trajectory fidelity.
  • Systematic debugging with test equipment (oscilloscope) identified root causes quickly.

Key Lesson: Hardware Matters

  • Software-only timing is unreliable on general-purpose OS. Use hardware peripherals (DMA, dedicated timers) for real-time constraints.
  • Jitter in one signal (PWM) propagates through the entire system (encoder ISR, controller stability).
  • Instrumenting the system (oscilloscope, CSV logging) is essential for debugging embedded control loops.
  • Small design choices early (hardware timer vs software loop) have massive impact on system reliability.
